## Architecture Principles

### Magento Module Structure

All custom code lives in `app/code/<Vendor>/<Module>/`. Never edit files under `vendor/`.

```
app/code/Vendor/Module/
├── registration.php          # Module registration
├── etc/
│   ├── module.xml            # Module declaration + sequence
│   ├── di.xml                # Dependency injection config
│   ├── frontend/routes.xml   # Frontend routes (if needed)
│   └── webapi.xml            # REST/GraphQL endpoints
├── Api/                      # Service contracts (interfaces only)
├── Api/Data/                 # Data interfaces
├── Model/                    # Implementations of Api interfaces
├── Model/ResourceModel/      # Database layer
├── Block/                    # View models (prefer ViewModels over Blocks)
├── ViewModel/                # Preferred: inject into templates
├── Controller/               # Action classes (one action per class)
├── Console/Command/          # CLI commands
├── Plugin/                   # Interceptors (prefer over rewrites)
├── Observer/                 # Event observers
├── Setup/Patch/Data/         # Data patches (idempotent)
├── Setup/Patch/Schema/       # Schema patches (prefer db_schema.xml)
├── db_schema.xml             # Declarative schema (preferred over InstallSchema)
├── view/frontend/            # Templates, layout XML, web assets
└── Test/
    ├── Unit/
    └── Integration/
```

### Core Architectural Rules

- **Service contracts first**: define interfaces in `Api/` before implementing in `Model/`
- **Prefer plugins over rewrites** (`around` → `before`/`after` → rewrite, in that order of preference)
- **Use `around` plugins sparingly** — they wrap the entire method and are hard to debug; use `before`/`after` when possible
- **No business logic in templates** — use ViewModels or Blocks; templates receive data, they don't compute it
- **No direct `ObjectManager` calls** — inject dependencies via constructor; the only exception is inside `registration.php` and static factories
- **Prefer `db_schema.xml`** over `InstallSchema`/`UpgradeSchema`; schema changes must be reversible
- **Data patches** must implement `DataPatchInterface` and `apply()` must be idempotent
- **Events vs Plugins**: use events for cross-cutting concerns (logging, notifications); use plugins for modifying behavior of a specific method

### Dependency Injection

```xml
<!-- etc/di.xml -->
<preference for="Vendor\Module\Api\ServiceInterface"
            type="Vendor\Module\Model\Service"/>

<type name="Vendor\Module\Model\Service">
    <arguments>
        <argument name="someConfig" xsi:type="string">value</argument>
    </arguments>
</type>
```

- Inject interfaces, not concrete classes
- Use `Proxy` for heavy dependencies that may not always be used
- Use `Factory` for objects that need to be created at runtime (non-shared)
- Virtual types (`<virtualType>`) to create specialized instances without new classes

### GraphQL (headless)

Since the frontend is headless React, prefer GraphQL over REST for storefront data:

- Schema files live in `etc/schema.graphqls`
- Resolvers in `Model/Resolver/`
- Resolver must implement `ResolverInterface` and return array matching schema
- Use `BatchContractResolverInterface` for N+1 avoidance

```bash
composer install
bin/magento setup:upgrade
bin/magento setup:di:compile
bin/magento setup:static-content:deploy -f
bin/magento cache:flush
```

```bash
bin/magento cache:clean                        # Clean specific cache types
bin/magento cache:flush                        # Flush all cache storage
bin/magento indexer:reindex                    # Reindex all
bin/magento module:enable Vendor_Module        # Enable a module
bin/magento setup:upgrade                      # Run after module changes
bin/magento setup:di:compile                   # Regenerate DI (required for prod)
```

### Developer mode

```bash
bin/magento deploy:mode:set developer
```

In developer mode: no DI compile needed, exceptions shown on screen, symlinks used for static assets.

### Code generation

Never commit files from `generated/`. These are regenerated by `setup:di:compile`.

## Coding Standards

### PHP

- Follow PSR-12
- Run php-cs-fixer before committing:
  ```bash
  vendor/bin/php-cs-fixer fix app/code/ --config=.php-cs-fixer.php --diff
  ```
- Static analysis with PHPStan (level 6 minimum):
  ```bash
  vendor/bin/phpstan analyse app/code/ --level=6
  ```
- Mess detection:
  ```bash
  vendor/bin/phpmd app/code/ text cleancode,codesize,controversial,design,naming,unusedcode
  ```
- Copy-paste detection:
  ```bash
  vendor/bin/phpcpd app/code/
  ```

### Magento-specific conventions

- Class names: `PascalCase`
- Method names: `camelCase`
- Constants: `UPPER_SNAKE_CASE`
- Table names: `vendor_module_entity` (snake_case)
- Event names: `vendor_module_event_name` (snake_case, all lowercase)
- Layout handles: `vendor_module_controller_action` (snake_case, all lowercase)
- Use `int|float|string|bool|array` type hints everywhere; avoid `mixed`
- Return type declarations on all methods

### Frontend (`frontend/`)

- React components: PascalCase files and function names
- Hooks: `use` prefix, camelCase
- API calls via `infrastructure/` layer only — no direct fetch in components
- Feature-based folder structure under `src/features/`

## Testing

### Test types and when to use each

| Type | Location | When |
|---|---|---|
| Unit | `dev/tests/unit/` or `Test/Unit/` | Pure PHP logic, no DB, no Magento bootstrap |
| Integration | `dev/tests/integration/` or `Test/Integration/` | DB interaction, DI container, multi-class flows |
| API Functional | `dev/tests/api-functional/` | REST/GraphQL endpoint contract testing |
| Acceptance (MFTF) | `dev/tests/acceptance/` | Full browser flows |
| Static | `dev/tests/static/` | Code style, copy-paste, legacy restrictions |

### Unit tests

```bash
vendor/bin/phpunit -c dev/tests/unit/phpunit.xml app/code/Vendor/Module/Test/Unit/
```

- Extend `PHPUnit\Framework\TestCase` (not Magento base class for pure unit tests)
- Use `dg/bypass-finals` to mock Magento final classes
- Mock interfaces, not concrete classes
- One assertion concept per test; descriptive test method names: `testShouldReturnZeroWhenCartIsEmpty`

Example:
```php
class PriceCalculatorTest extends TestCase
{
    private PriceCalculator $calculator;

    protected function setUp(): void
    {
        $tax = $this->createMock(TaxCalculatorInterface::class);
        $tax->method('calculate')->willReturn(0.0);
        $this->calculator = new PriceCalculator($tax);
    }

    public function testShouldApplyDiscountCorrectly(): void
    {
        $result = $this->calculator->applyDiscount(100.0, 10);
        $this->assertSame(90.0, $result);
    }
}
```

### Integration tests

```bash
vendor/bin/phpunit -c dev/tests/integration/phpunit.xml \
  app/code/Vendor/Module/Test/Integration/
```

- Extend `Magento\TestFramework\TestCase\AbstractController` for controller tests
- Use `@magentoDbIsolation enabled` to roll back DB changes per test
- Use `@magentoDataFixture` to load fixtures
- Use `@magentoConfigFixture` to set config values per test

### Static analysis (Magento-specific)

```bash
# Run Magento's own static checks
vendor/bin/phpunit -c dev/tests/static/phpunit.xml
```

### MFTF acceptance tests

```bash
vendor/bin/mftf run:test CheckoutGuestTest
vendor/bin/mftf run:group checkout
```

Tests live in `dev/tests/acceptance/tests/functional/`.

### `ci.yml` — Pull Request checks

```yaml
name: CI

on:
  pull_request:
    branches: [master, develop]
  push:
    branches: [master]

env:
  PHP_VERSION: "8.3"

jobs:
  code-quality:

      - uses: actions/checkout@v4

          tools: composer:v2
          coverage: none

        uses: actions/cache@v4
        with:
          path: ~/.composer/cache
          key: composer-${{ hashFiles('composer.lock') }}
          restore-keys: composer-

      - name: Install dependencies
        run: composer install --prefer-dist --no-progress --optimize-autoloader

      - name: PHP CS Fixer

      - name: PHPStan
        run: vendor/bin/phpstan analyse app/code/ --level=6 --no-progress

      - name: PHPMD
        run: vendor/bin/phpmd app/code/ github cleancode,codesize,controversial,design,naming,unusedcode

      - name: PHPCPD
        run: vendor/bin/phpcpd app/code/ --min-lines=5

  unit-tests:
    name: Unit Tests
    runs-on: ubuntu-latest
    needs: code-quality
    steps:
      - uses: actions/checkout@v4

      - uses: shivammathur/setup-php@v2

          php-version: ${{ env.PHP_VERSION }}

          path: ~/.composer/cache

            -c dev/tests/unit/phpunit.xml \

  integration-tests:

          --health-timeout=5s

          - 3306:3306

        image: opensearchproject/opensearch:2.19.5
        env:
          discovery.type: single-node
          DISABLE_SECURITY_PLUGIN: "true"
        ports:
          - 9200:9200

    steps:
      - uses: actions/checkout@v4

      - uses: shivammathur/setup-php@v2
        with:
          php-version: ${{ env.PHP_VERSION }}
          extensions: bcmath, ctype, curl, dom, gd, hash, intl, mbstring, openssl, pdo_mysql, simplexml, soap, xsl, zip
          tools: composer:v2
          coverage: none

      - name: Cache Composer dependencies
        uses: actions/cache@v4
        with:
          path: ~/.composer/cache
          key: composer-${{ hashFiles('composer.lock') }}
          restore-keys: composer-

      - name: Install dependencies

      - name: Prepare integration test config

        run: |
          vendor/bin/phpunit \
            -c dev/tests/integration/phpunit.xml \

    runs-on: ubuntu-latest
```

### `deploy-staging.yml` — Deploy to staging on merge to `develop`

```yaml
name: Deploy Staging

on:
  push:
    branches: [develop]

jobs:
  deploy:
    name: Deploy to Staging
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.STAGING_HOST }}
          username: ${{ secrets.STAGING_USER }}
          key: ${{ secrets.STAGING_SSH_KEY }}
          script: |
            cd /var/www/html/dm3dtech
            git pull origin develop
            composer install --no-dev --optimize-autoloader --no-interaction
            bin/magento maintenance:enable
            bin/magento setup:upgrade --keep-generated
            bin/magento setup:di:compile
            bin/magento setup:static-content:deploy pt_BR en_US -f
            bin/magento cache:flush
            bin/magento maintenance:disable
```

### `deploy-production.yml` — Deploy to production on release tag

```yaml
name: Deploy Production

on:
  push:
    tags:
      - 'v[0-9]+.[0-9]+.[0-9]+'

jobs:
  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    environment: production
    steps:
      - uses: actions/checkout@v4

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1

          host: ${{ secrets.PROD_HOST }}
          username: ${{ secrets.PROD_USER }}
          key: ${{ secrets.PROD_SSH_KEY }}

            git fetch --tags
            git checkout ${{ github.ref_name }}
            composer install --no-dev --optimize-autoloader --no-interaction
            bin/magento maintenance:enable
            bin/magento setup:upgrade --keep-generated
            bin/magento setup:di:compile
            bin/magento setup:static-content:deploy pt_BR en_US -f
            bin/magento cache:flush
            bin/magento maintenance:disable
```

### Branch strategy

```
master          ← production; tagged releases only
develop         ← staging; merges from feature branches
feature/*       ← new features (PR → develop)
```

## Security

- Never commit `app/etc/env.php` — it contains DB credentials and encryption key
- Never commit `auth.json` — it contains Magento Marketplace credentials
- Both are in `.gitignore`; verify with `git status` before committing
- Use GitHub secrets for all credentials in CI/CD
- Run `bin/magento security:check` periodically for known vulnerabilities
- Apply patches via `vendor/bin/magento-patches apply` (Magento Quality Patches)

## Performance

- Always run `setup:di:compile` before deploying (30–40% performance gain)
- Use `setup:static-content:deploy` with specific locales (`pt_BR en_US`) to avoid deploying unused locales
- Configure Varnish for full-page cache in production
- Use `--keep-generated` flag during deploys to avoid recompiling if code unchanged
- Index on deploy: `bin/magento indexer:reindex` or configure indexers to update on schedule

## Headless Frontend (`frontend/`)

The `frontend/` directory is a standalone React SPA. It communicates with Magento exclusively via GraphQL.

```bash
cd frontend
npm install
npm run dev        # Local dev server
npm run build      # Production build
npm run preview    # Preview production build locally
```

- GraphQL endpoint: configured via `VITE_GRAPHQL_URL` env var
- API calls must go through `src/infrastructure/` — never call fetch/axios directly in components or hooks
- Domain logic in `src/domain/` — pure TypeScript, no React dependencies
- Business logic in `src/application/` — hooks and context that coordinate domain + infrastructure
- UI-only components in `src/components/` — receive data via props, no API calls

## Mage-OS Updates

Mage-OS packages ship from `https://repo.mage-os.org/` and follow Magento 2's versioning (`2.4.x`). The core metapackage is `mage-os/product-community-edition`.

### Check for available updates

```bash
# Show outdated Mage-OS and Magento packages
composer outdated 'mage-os/*' 'magento/*'

# Show all outdated packages including extensions
composer outdated
```

### Update Mage-OS core

```bash
# Bump the version constraint in composer.json, then:
composer update mage-os/product-community-edition --with-all-dependencies

# Or update everything within current constraints:
composer update
```

After any composer update that touches Mage-OS/Magento packages:

```bash
bin/magento maintenance:enable
bin/magento setup:upgrade
bin/magento setup:di:compile
bin/magento setup:static-content:deploy pt_BR en_US -f
bin/magento cache:flush
bin/magento maintenance:disable
```

### Dependency version audit

The project includes `mage-os/composer-dependency-version-audit-plugin`, which warns when installed package versions conflict with Mage-OS version requirements. It runs automatically on `composer install`/`update`. If it reports conflicts, resolve them before deploying.

### Quality Patches

Apply official Magento Quality Patches (MQP) via the patches tool:

```bash
# List all available patches and their status
vendor/bin/magento-patches status

# Apply a specific patch
vendor/bin/magento-patches apply MDVA-XXXXX

# Apply all recommended patches
vendor/bin/magento-patches apply --all

# Revert a patch
vendor/bin/magento-patches revert MDVA-XXXXX
```

Patches are tracked in `composer.lock` via the `mage-os/magento-composer-installer` plugin.

### GitHub Actions — automated update check

Add this workflow to catch available updates on a schedule:

```yaml
# .github/workflows/dependency-update-check.yml
name: Dependency Update Check

on:
  schedule:
    - cron: '0 8 * * 1'  # Every Monday at 08:00 UTC
  workflow_dispatch:

jobs:
  check-updates:
    name: Check for Mage-OS updates
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: shivammathur/setup-php@v2
        with:
          php-version: "8.3"
          tools: composer:v2
          coverage: none

      - name: Install dependencies
        run: composer install --prefer-dist --no-progress

      - name: Check outdated packages
        run: composer outdated --direct --format=json | tee outdated.json

      - name: Post summary
        run: |
          echo "## Outdated packages" >> $GITHUB_STEP_SUMMARY
          cat outdated.json | \
            python3 -c "
          import json, sys
          data = json.load(sys.stdin)
          pkgs = data.get('installed', [])
          if not pkgs:
              print('All packages are up to date.')
          else:
              print('| Package | Current | Latest |')
              print('|---|---|---|')
              for p in pkgs:
                  print(f\"| {p['name']} | {p['version']} | {p['latest']} |\")
          " >> $GITHUB_STEP_SUMMARY
```

### Update checklist

Before updating Mage-OS version on staging:

- [ ] Read the [Mage-OS release notes](https://mage-os.org/distribution/) and Magento release notes for the target version
- [ ] Check compatibility of `imaginationmedia/correios` with the new version
- [ ] Check compatibility of all `require-dev` tools (phpunit, phpstan, MFTF)
- [ ] Run `composer update` on a feature branch, not directly on `develop`
- [ ] Run the full CI pipeline (unit + integration + static)
- [ ] Deploy to staging and run smoke tests (checkout, search, cart)
- [ ] Check OpenSearch index compatibility — reindex if mappings changed
- [ ] Verify `setup:db:status` reports no pending upgrades before disabling maintenance mode

## Common pitfalls to avoid

- Do not use `$_objectManager` or `\Magento\Framework\App\ObjectManager::getInstance()` in custom code
- Do not override layout XML from core — use layout handles correctly; add your own handles
- Do not use `Zend_Db_Expr` raw SQL in models — use the collection/resource model API
- Do not call `setup:upgrade` in production without `maintenance:enable` first
- Do not run `setup:di:compile` in developer mode (it's unnecessary and slows down iteration)
- Do not place business logic in `Block` classes — use ViewModels
- Do not create hard dependencies on third-party modules in `module.xml` unless required; use `sequence` for soft ordering
- Do not store sensitive data in Magento config — use environment variables and `env.php`
